Lesson 0006 - 25 minutes

Semantic search starts with the chunk, not the vector.

Embeddings make language searchable, but your retrieval quality depends on what you choose to embed.

The One Idea

An embedding is a numeric representation of a piece of text. Similar text lands near similar text, which lets your app search by meaning instead of exact keywords.

The engineering decision is the chunk. A chunk should be small enough to retrieve precisely and large enough to preserve the idea a user needs.

1. Source

Start with a document, ticket, policy page, meeting note, or product spec.

2. Chunk

Split the source into meaningful units with stable IDs and metadata.

3. Embed

Convert each chunk into a vector using an embedding model.

4. Store

Save vectors plus text, source, permissions, and other metadata.

5. Retrieve

Embed the query, search nearest chunks, then filter by product rules.

Chunk Record Shape

type Chunk = {
  id: string;
  sourceId: string;
  title: string;
  text: string;
  url?: string;
  sectionPath: string[];
  updatedAt: string;
  audience: "public" | "internal" | "restricted";
};

Why Metadata Matters

  • sourceId gives citations a stable target.
  • sectionPath helps users inspect where evidence came from.
  • updatedAt lets you downrank stale content.
  • audience supports access-control filtering before generation.

Production Habit

Never treat the vector database as the source of truth. Store enough metadata to re-fetch, re-index, cite, delete, and permission-check every chunk.

Build Assignment

Create a tiny corpus for the Product Triage Assistant: five fake product docs, each with two to four chunks.

  • Give every chunk a stable ID.
  • Add source title, section path, audience, and updated date.
  • Write three user questions that should retrieve one specific chunk.

Ask the agent to review whether each chunk is too broad, too tiny, or missing useful metadata.
