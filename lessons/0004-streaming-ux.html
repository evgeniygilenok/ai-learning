Lesson 0004 - 25 minutes

Streaming is a product decision, not just an API option.

Streaming improves perceived speed, but it also changes UI state, cancellation, validation, and error handling.

The One Idea

Users do not experience "latency" as one number. They experience time to first signal, progress, control, and whether the final answer lands cleanly. Streaming helps with the first two, but the application still needs a final validation step.

For structured outputs, a common pattern is to stream user-facing explanation while treating the final structured object as complete only after the response is done.

Streaming State Machine

idle
  -> submitting
  -> streaming
  -> finalizing
  -> complete

error can happen at:
- submit failed
- stream interrupted
- final validation failed
- user cancelled

What Each State Owns

  • submitting: disable duplicate sends.
  • streaming: append partial text and show cancel.
  • finalizing: validate structured data and usage metadata.
  • complete: enable save, copy, and next action.

Four Rules For Streaming AI UX

1. Show first signal fast

A placeholder is weaker than visible progress. Even a short status line helps the user trust the system is working.

2. Keep cancellation real

Cancel should stop the request, stop rendering, and leave the UI in a recoverable state.

3. Validate at the end

Partial text is not a valid final object. Treat final structured data as complete only after final validation.

4. Design the unhappy path

Interrupted streams need a useful recovery message, not a half-finished answer pretending to be done.
