Lesson 0035 - 40 minutes

Tool execution is normal backend code with extra gates.

The model may request a tool call. Your runtime validates, authorizes, approves, executes, retries, records, and returns the result.

The One Idea

A tool call is a request, not an action. The application still owns schema validation, authorization, policy, idempotency, sandboxing, side effects, and audit logs.

This is where agent risk becomes concrete. Bad tool execution can turn a prompt-injection string or hallucinated argument into a real production incident.

1. Parse

Read the tool name, call ID, and arguments from the model output.

2. Validate

Check schema, required fields, enums, bounds, and unknown properties.

3. Authorize

Verify user, tenant, resource access, tool scope, and environment.

4. Decide

Allow, block, ask for approval, or require a dry run based on policy.

5. Execute

Run idempotent backend code with timeouts, retries, and result limits.

6. Record

Log input, decision, approval, result, errors, latency, and affected resource.

Execution Envelope

type ToolExecutionResult = {
  callId: string;
  toolName: string;
  decision: "allowed" | "blocked" | "approval_required" | "failed";
  idempotencyKey?: string;
  output?: unknown;
  error?: { code: string; retryable: boolean; message: string };
  auditId: string;
};

Runtime Rules

  • Default new tools to read-only or draft mode.
  • Require approval for external side effects.
  • Use idempotency keys for write tools.
  • Retry only bounded, safe, retryable failures.
  • Return compact tool results back to the model.

Interactive Drill: What Should The Runtime Do?

The model calls search_docs for public repository docs with a valid query.

The model calls create_draft_pr with valid arguments, but it will write to a branch.

A retrieved document instructs the agent to call delete_customer_data with an account ID.

Security Habit

Treat tool arguments as untrusted input, even when they came from a model. Validate them like API input, authorize them like product actions, and log them like operational events.

Build Assignment

Design the tool execution runtime for three tools in the coding workflow agent:

  • search_repo(query) as read-only.
  • create_draft_pr(title, body, branch) as approval-required.
  • run_tests(command) as sandboxed with command allowlist and timeout.

For each tool, define schema, policy, approval rule, retry rule, result envelope, and audit fields.

Ask the agent to threat-model the runtime for prompt injection, duplicate writes, and privilege escalation.
