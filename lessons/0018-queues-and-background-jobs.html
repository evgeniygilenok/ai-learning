Lesson 0018 - 30 minutes

Move slow AI work out of the request path.

Queues and background jobs keep AI products responsive when ingestion, evals, retries, or long workflows take too long.

The One Idea

Some AI work belongs in the foreground: short streaming replies, quick structured extraction, and user-visible tool decisions.

Other work belongs in the background: document ingestion, embedding updates, eval suites, long agent runs, retries, and report generation. The product should show status instead of making the user wait blindly.

Ingestion Jobs

Parse documents, chunk them, embed them, and update search indexes without blocking users.

Eval Jobs

Run golden cases after prompt, retrieval, model, or tool-policy changes.

Workflow Jobs

Continue multi-step agent work with checkpoints, approvals, retries, and resumable state.

Maintenance Jobs

Clean old traces, refresh caches, enforce retention, and rebuild stale indexes.

Job Record Shape

type AiJob = {
  id: string;
  tenantId: string;
  type: "ingest" | "eval" | "agent_run" | "cleanup";
  status: "queued" | "running" | "waiting_for_approval" | "failed" | "complete";
  attempts: number;
  idempotencyKey: string;
  traceId?: string;
  createdAt: string;
};

Queue Design Rules

  • Use idempotency keys so retries do not duplicate side effects.
  • Preserve tenant and authorization context with the job.
  • Expose user-visible status for long work.
  • Record traces for failures, retries, and approvals.

Interactive Drill: Foreground Or Background?

Re-embed 400 updated product documents after a docs import.

Stream a short answer while the user watches the chat window.

Run a full retrieval eval suite before merging a retriever change.

Reliability Habit

Every queued AI job should answer three questions: can it retry safely, can a user understand its status, and can an engineer debug its failure?

Build Assignment

Design the background job model for your Enterprise Knowledge Assistant.

  • Define one ingestion job and one eval job.
  • Add status values and retry rules for each.
  • Define what the user sees while the job runs.
  • List the trace fields needed to debug a failed job.

Ask the agent to review your jobs for idempotency, tenant isolation, and retry safety.
