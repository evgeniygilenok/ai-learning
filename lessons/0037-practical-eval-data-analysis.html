Lesson 0037 - 40 minutes

Read eval data like an engineer, not like a scoreboard.

The practical Python/data gap is small but important: you need to group failures, compare runs, and turn raw cases into a decision.

The One Idea

An eval result is useful only when it changes an engineering decision. Your job is to find failure clusters, not celebrate a single aggregate score.

1. Normalize

Every case should have an id, input, expected behavior, observed behavior, pass/fail, tags, latency, cost, and trace link.

2. Group

Cluster failures by root cause: retrieval miss, bad context, schema failure, unsafe tool request, judge ambiguity, or answer style.

3. Compare

Compare baseline versus candidate by quality, latency, cost, and new regressions.

4. Decide

Ship, block, review, or run a smaller experiment based on risk and evidence.

JSONL Shape

{"id":"rag-018","tags":["acl","answerability"],"input":"Can contractors view payroll policy?","expected_sources":["policy-access-03"],"observed_sources":["payroll-general-01"],"passed":false,"failure":"retrieval_acl","latency_ms":1840,"cost_usd":0.012,"trace_id":"tr_abc123"}

Report Shape

Run: reranker-v2
Pass rate: 24/30
Regressions: 2
Top failure cluster: retrieval_acl
Decision: block release
Next fix: enforce ACL filter before rerank

Build Assignment

Create a tiny eval analysis script for the portfolio.

  • Read a JSONL file of at least 20 eval cases.
  • Group failures by failure and tags.
  • Compare a baseline run and a candidate run.
  • Generate docs/eval-analysis-report.md.
  • Include one release decision: ship, block, or review.

Ask the agent to help design the smallest useful JSONL schema before writing code.
