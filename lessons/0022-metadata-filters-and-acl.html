Lesson 0022 - 30 minutes

Metadata filters are the guardrails around semantic search.

Vector similarity finds likely meaning. Metadata and ACL filters decide which evidence is allowed, fresh, and relevant to the user.

The One Idea

Search relevance is not enough. A result must also satisfy tenant, role, source, region, freshness, product area, and document-state constraints before it enters the model context.

Access-control filtering should happen before generation, not as an instruction asking the model to ignore forbidden chunks.

Tenant Filter

Every search is scoped to the current tenant or explicitly shared corpus.

ACL Filter

Chunk permissions are checked against the user's role, groups, and entitlement.

Freshness Filter

Expired, deleted, draft, or stale content is excluded or downranked.

Domain Filter

Product area, document type, region, and source system narrow the search space.

Filter Shape

type RetrievalFilter = {
  tenantId: string;
  allowedGroups: string[];
  region?: "us" | "ca" | "eu";
  sourceTypes: string[];
  includeDrafts: false;
  updatedAfter?: string;
};

Design Rule

Combine semantic ranking with metadata constraints. If a chunk fails the filter, it should not be shown to the model, cited, cached, or logged as user-visible evidence.

Interactive Drill: Safe Filter?

Search filters by tenant ID and allowed groups before returning text.

Search returns all chunks, then asks the model to hide forbidden ones.

Cache key includes query text but not tenant or ACL context.

Build Assignment

Write the retrieval filter contract for your Enterprise Knowledge Assistant.

  • Define required metadata fields on every chunk.
  • Write the filter object for a normal user and an admin.
  • Define what happens to stale or deleted content.
  • Add one eval case for cross-tenant leakage.

Ask the agent to review your filter as a data isolation reviewer.
