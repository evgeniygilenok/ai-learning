Lesson 0010 - 35 minutes

A useful agent is a controlled workflow, not a loose chatbot.

Agents become production-shaped when state, tools, approvals, retries, and stop conditions are explicit.

The One Idea

An agent can choose steps and use tools, but the application should still own the workflow boundary. Treat autonomy as a dial, not a switch.

For senior AI engineering work, the strongest pattern is often a state machine where the model helps inside each state and product code decides allowed transitions.

1. Intake

Read the task, classify risk, and decide whether enough information exists.

2. Clarify

Ask a targeted question instead of guessing when the task is underspecified.

3. Plan

Produce a small, inspectable plan with expected files, tools, or evidence.

4. Dry Run

Call read-only or draft tools and collect results without external side effects.

5. Human Review

Pause before risky writes, customer communication, money movement, or irreversible changes.

6. Execute

Run approved actions, capture trace data, and produce a final summary.

Workflow Sketch

intake
  -> clarify when required fields are missing
  -> plan when task is understood
  -> dry_run for read-only tools
  -> human_review for risky actions
  -> execute after approval
  -> complete or recoverable_error

Boundary Rules

  • The model may propose a next state.
  • The app validates whether that transition is allowed.
  • Risky tools require explicit approval.
  • Every run has a max step count and a stop condition.
  • Tool results become state, not hidden memory.

Interactive Drill: Pick The Next State

The user says "fix checkout" with no error, repo context, or expected behavior.

The agent drafted a customer email about a payment incident.

The agent needs to inspect public docs and summarize matching sections.

Portfolio Angle

A controlled workflow agent is easier to defend in interviews than a demo that just loops until it looks done. You can explain states, permissions, recovery, traces, and why each tool is scoped.

Build Assignment

Design a dry-run coding workflow agent for a Jira-style ticket.

  • Define six states from intake to final checklist.
  • List the tools available in each state.
  • Mark each tool as read-only, draft, or side-effecting.
  • Add two approval gates and two stop conditions.

Ask the agent to critique the workflow for excessive agency and missing recovery paths.
