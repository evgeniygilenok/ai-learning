Lesson 0027 - 35 minutes

Enterprise RAG is search, security, and evaluation glued together.

The system design question is not "vector database or not?" It is how evidence becomes reliable, permitted, cited context.

The One Idea

A senior enterprise RAG design starts with source authority and access boundaries, then adds ingestion, retrieval, reranking, answer generation, evals, and observability.

Source Layer

Docs, tickets, wikis, policies, permissions, freshness, and deletion events.

Retrieval Layer

Chunking, embeddings, hybrid search, metadata filters, reranking, and context budgets.

Answer Layer

Grounded generation, citation rules, refusal behavior, and structured response fields.

Control Layer

Evals, traces, tenant isolation, feedback, retention, and release gates.

Whiteboard Flow

sources -> ingestion jobs
  -> chunks + metadata + vectors
  -> filtered hybrid retrieval
  -> reranked context packet
  -> grounded answer with citations
  -> trace + eval + feedback loop

Tradeoffs To Defend

  • Vector DB versus Postgres plus pgvector.
  • Chunk size and source diversity.
  • Freshness versus indexing cost.
  • Recall versus context noise.

Interactive Drill: Design Priority

Users from Tenant A are seeing Tenant B policy snippets.

Answers are well-written but cite irrelevant docs.

Team cannot tell whether chunking changes improved quality.

Build Assignment

Prepare a five-minute enterprise RAG whiteboard answer.

  • Draw source, ingestion, retrieval, answer, and observability layers.
  • Name three failure modes and mitigations.
  • Define two retrieval metrics and one answer metric.
  • Explain tenant isolation in one minute.

Ask the agent to push back on your architecture like an interviewer.
