Lesson 0028 - 35 minutes

A coding copilot is a context engine with a careful action boundary.

The challenge is not generating code. It is choosing context, proposing changes, testing safely, and earning developer trust.

The One Idea

A coding copilot needs repository context, task understanding, change planning, edit generation, test selection, and human review. Autonomy should increase only after evidence and safety gates are strong.

Context Builder

Find relevant files, symbols, tests, docs, and recent changes without flooding the model.

Planner

Explain intended files, risks, and tests before editing.

Executor

Apply scoped patches, run checks, capture failures, and preserve user changes.

Reviewer

Summarize diffs, risks, test results, and places needing human judgment.

Control Flow

ticket -> clarify
  -> retrieve repo context
  -> plan
  -> propose patch
  -> run tests
  -> review summary
  -> human approves PR

Safety Rules

  • Never overwrite unrelated user changes.
  • Prefer small, reviewable patches.
  • Run targeted tests before broad tests.
  • Ask before destructive Git operations.

Interactive Drill: Copilot Boundary

Read files and propose a patch plan.

Run a destructive Git reset to clean the repo.

Open a PR that triggers external CI and notifications.

Build Assignment

Whiteboard a coding copilot for senior interview practice.

  • Define the context-selection strategy.
  • Define edit, test, and review states.
  • List three tool permissions and two approval gates.
  • Choose evals for patch quality and test selection.

Ask the agent to critique your design for unsafe autonomy and missing developer trust signals.
