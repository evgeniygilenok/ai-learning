Agent Runtime Reference

A compact field guide for the five runtime concerns behind useful agents: orchestration, state, memory, evaluation, and tool execution.

Orchestration

Who controls the next step?

Choose explicit workflow, manager-with-tools, handoff, or background run. The key decision is ownership of control and the final user-facing answer.

Review Lesson 0031

State

What resumes this run?

Track current state, facts, pending decisions, budgets, retries, approvals, and trace pointers. State is the run ledger, not long-term memory.

Review Lesson 0032

Memory

What should survive later?

Persist only useful, scoped, sourced, visible, and expiring memories. Reject secrets, stale guesses, and untrusted instructions.

Review Lesson 0033

Evaluation

How do we know it worked?

Score final output and trace path: state sequence, tools, approvals, memory writes, recovery, budget, and safety policy.

Review Lesson 0034

Tools

What side effects are allowed?

Treat tool calls as untrusted requests. Validate, authorize, decide, execute, retry, record, and return compact results.

Review Lesson 0035

Runtime Loop

load state
choose orchestration branch
build model context from state + allowed memory
run model
validate output or tool call
execute allowed tools through policy gates
persist state patch and trace
pause for approval, continue, or stop
run evals over representative traces

Default Safety Posture

  • Read-only before write-capable.
  • Draft before publish.
  • Approval before external side effect.
  • State before action.
  • Trace before eval automation.
  • Exact assertions before judge models.

Design Matrix

Concern Owns Must Include Failure Signal
Orchestration Control flow and ownership Pattern, owner, allowed tools, next state, stop conditions The agent loops, hands off vaguely, or nobody owns the final answer.
State Current run snapshot Run ID, current state, facts, approvals, budgets, trace IDs A paused or failed run cannot resume without replaying the whole conversation.
Memory Future-facing retained knowledge Scope, kind, source, visibility, confidence, expiration The system saves secrets, stale guesses, or irrelevant transcript fragments.
Evaluation Repeatable quality checks Dataset, trace assertions, forbidden tools, required approvals, rubric A prompt or tool change feels better but cannot be compared to prior behavior.
Tool Execution External data and side effects Schema, auth, policy, approval, idempotency, retries, audit A model request turns directly into an unsafe or duplicate action.

TypeScript Skeleton

type AgentRuntime = {
  orchestration: OrchestrationDecision;
  state: AgentRunState;
  retrievedMemory: AgentMemory[];
  toolPolicy: ToolPolicy;
  trace: RuntimeTrace;
  evalCase?: AgentEvalCase;
};

type RuntimeDecision =
  | { kind: "continue"; statePatch: Partial<AgentRunState> }
  | { kind: "approval_required"; approvals: ToolApproval[] }
  | { kind: "blocked"; reason: string }
  | { kind: "done"; finalOutput: unknown };
