Lesson 0024 - 30 minutes

Agent reliability is mostly recovery design.

Multi-step workflows fail in normal ways: missing info, bad tool arguments, timeouts, partial success, and user rejection.

The One Idea

A workflow agent should not improvise forever. It should classify failures, retry only safe operations, checkpoint progress, and ask for help when uncertainty or risk is high.

Recovery behavior belongs in application state, not hidden in a prompt.

Validation Failure

Tool args or structured output fail schema checks. Repair once, then stop or escalate.

Transient Failure

Provider timeout, rate limit, or network issue. Retry with backoff when idempotent.

Business Failure

Missing approval, blocked policy, or insufficient permissions. Ask or stop.

Partial Success

Some steps completed. Resume from checkpoints instead of starting over.

Recovery Policy

type RecoveryPolicy = {
  maxSteps: number;
  maxRetriesPerTool: number;
  retryOnlyIfIdempotent: true;
  checkpointAfter: string[];
  requireHumanFor: ["permission_denied", "side_effect"];
};

Design Rule

Every agent state should define what happens on success, validation failure, transient failure, user rejection, and timeout.

Interactive Drill: Retry Or Stop?

A read-only docs lookup times out once.

A tool returns permission denied for the current user.

The agent is about to repeat a side-effecting action with no idempotency key.

Build Assignment

Add recovery rules to your controlled workflow agent.

  • Define max step count and max retries.
  • Choose checkpoints for plan, dry run, review, and execution.
  • List which tools are safe to retry.
  • Define user-facing messages for stopped and recoverable failures.

Ask the agent to look for retry loops, duplicate side effects, and missing checkpoints.
