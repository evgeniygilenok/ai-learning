Lesson 0001 - 20 minutes

The AI engineering map: wrap probabilistic models in product contracts.

You already know how to build software. The new skill is designing the boundaries around model behavior so the product is useful, measurable, and safe enough to ship.

The One Idea

Traditional product code is mostly deterministic: given the same input and state, you expect the same output. AI features include a probabilistic model call, so the engineering job changes.

AI engineering is not "find a magic prompt." It is the discipline of surrounding model calls with contracts: what goes in, what context is allowed, what shape comes out, what actions may happen, how quality is measured, and what risks are blocked.

The Six Contracts

1. Input

What user intent, product state, and constraints enter the model call?

2. Context

What retrieved docs, history, files, or examples are allowed into the context window?

3. Model

Which model, parameters, latency target, cost budget, and fallback path are used?

4. Output

Does the answer need prose, JSON, citations, confidence, or a refusal state?

5. Action

Which tools can the model request, and what does your app execute or require approval for?

6. Quality and Safety

How do you test, trace, monitor, and constrain the system before and after release?

Why The Roadmap Starts Small

Your draft plan is strong, but the order should avoid a common trap: learning RAG, agents, evals, and security as separate theory islands.

Instead, build a small product loop first, then improve the same loop. That gives every new concept a place to live.

First Portfolio Spine

Use one cumulative system:

  • Chat slice: product triage assistant with streaming and structured output.
  • RAG slice: assistant grounded in a docs corpus with citations and retrieval tests.
  • Agent slice: controlled workflow that proposes actions and asks for approval.

How The Same Product Evolves

Stage User Request Engineering Move Proof It Works
Thin LLM app "Turn this vague feature idea into a product brief." Prompt, model call, streaming UI, structured output schema. Saved examples produce complete briefs with valid JSON.
RAG app "Use our internal product docs and cite your sources." Chunk docs, embed them, retrieve relevant context, answer with citations. Retrieval tests find the right source chunks for known questions.
Agent workflow "Prepare the Jira implementation plan and test checklist." State machine, tools, memory, approval gates, traceable actions. The agent stops for approval before side effects and leaves an audit trail.

Practice: Upgrade A Weak Prompt

Weak prompt:

You are helpful. Summarize this feature request and say what to do next.

Before revealing the answer, say out loud what contract is missing. Hint: there are at least four.

Stronger prompt skeleton:

Task: Convert a vague feature request into a product triage brief. Audience: Senior product and engineering reviewers. Use only the user's request and the provided product context. If required information is missing, ask one focused follow-up question instead of inventing details. Return valid JSON matching: { "summary": "one sentence", "assumptions": ["short bullets"], "riskLevel": "low | medium | high", "risks": ["product, technical, or compliance risks"], "nextQuestion": "one question", "suggestedNextAction": "one practical action" }

This adds input boundaries, context boundaries, output shape, uncertainty behavior, and review-oriented product framing.

Your First Build Target

Build the smallest useful version of a Product Triage Assistant. The first version does not need RAG, agents, or a database.

  • A textarea accepts a feature request.
  • The model streams a short response for the user.
  • The app also stores a structured JSON brief.
  • Five saved test cases check that the JSON shape is valid and the next question is useful.

This is enough to start learning like an AI engineer because every later topic improves a visible system.
