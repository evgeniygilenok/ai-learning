Lesson 0011 - 30 minutes

Agent memory is persisted state with a product reason.

Do not let "memory" become a vague bucket. Decide what should be saved, why it matters, who can see it, and when it expires.

The One Idea

In an agentic workflow, memory is not magic recall. It is application state that survives long enough to improve the next step or the next run.

The senior engineering move is to separate working state, durable facts, user preferences, and audit history. Each one has different privacy, retention, and retrieval rules.

Working State

Temporary state for the current run: ticket ID, current plan, tool results, missing fields, and step count.

Durable Facts

Stable project knowledge such as service ownership, docs locations, architecture decisions, or team conventions.

User Preferences

Explicit preferences that shape future behavior, such as preferred output format or review strictness.

Audit History

A record of actions, approvals, tool calls, and outcomes for debugging, compliance, and accountability.

Memory Record Shape

type MemoryRecord = {
  id: string;
  tenantId: string;
  subject: "run" | "project" | "user" | "audit";
  text: string;
  source: "explicit_user" | "tool_result" | "system_event";
  visibility: "private" | "team" | "admin";
  expiresAt?: string;
  createdAt: string;
};

Memory Rules

  • Only save memory with a named product use.
  • Prefer explicit user-provided preferences over inferred ones.
  • Keep secrets and transient credentials out of memory.
  • Store tool results as state with provenance, not as hidden vibes.

Interactive Drill: Save Or Do Not Save?

The user says: "For this project, always produce PR checklists in markdown."

A temporary OAuth token appears in a tool error log.

A tool confirms that ticket PAY-431 belongs to the billing service owned by Team Ledger.

Security Habit

Memory expands blast radius. Every persisted fact needs a visibility boundary, a deletion path, and a reason it should outlive the current request.

Build Assignment

Add memory design to the dry-run coding workflow agent from Lesson 0010.

  • List five things the agent may store.
  • List five things it must never store.
  • Define a retention policy for working state and audit history.
  • Decide which memories are visible to user, team, or admin roles.

Ask the agent to review your memory rules for privacy risk and missing product purpose.
