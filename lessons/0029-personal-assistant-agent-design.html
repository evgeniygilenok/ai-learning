Lesson 0029 - 35 minutes

A personal assistant agent is mostly permissions and memory design.

Calendar, tasks, messages, and knowledge tools are useful only when scoped by consent, context, and reversibility.

The One Idea

A personal assistant sits close to private life and work. The product must separate suggestions from actions, short-term context from durable memory, and read-only tools from side-effecting tools.

Channels

Chat, email, calendar, task manager, docs, and notifications all have different consent surfaces.

Memory

Store explicit preferences and durable facts, not stray private details from every conversation.

Tools

Read calendars and tasks broadly; require approval before sending, booking, deleting, or purchasing.

Feedback

Let users correct memory, undo drafts, inspect actions, and tune autonomy levels.

Autonomy Ladder

suggest -> draft -> schedule tentative
  -> ask approval
  -> execute
  -> log action
  -> allow correction or undo

Risk Rule

The more personal, external, costly, or irreversible the action, the more explicit approval and auditability the assistant needs.

Interactive Drill: Autonomy Level

Summarize today's calendar and suggest focus blocks.

Send a reschedule email to a client.

Delete all tasks tagged "old".

Build Assignment

Whiteboard a personal assistant agent.

  • Define three tools and mark read-only or side-effecting.
  • Define memory rules for preferences, contacts, and tasks.
  • Create an approval policy for messages, calendar, and deletion.
  • Choose one eval for usefulness and one eval for unsafe autonomy.

Ask the agent to review your design for privacy and consent gaps.
