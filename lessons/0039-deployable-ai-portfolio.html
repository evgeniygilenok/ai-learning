Lesson 0039 - 45 minutes

Make your AI portfolio runnable, not just impressive.

A reviewer trusts a project faster when setup, secrets, seed data, health checks, and eval smoke tests are boringly clear.

The One Idea

Deployment proof does not have to mean a fully public production SaaS. It means the project is reproducible, observable, and safe to try without private context or fragile setup magic.

Run path

One clean path: local dev, Docker Compose, or hosted demo. Do not make reviewers reverse-engineer setup.

Demo data

Seed docs, fake users, fake tenants, fake tickets, and saved traces make the demo inspectable.

Config safety

Document env vars, separate local/staging/prod, and keep secrets out of code and logs.

Operational proof

Expose health, recent runs, eval smoke tests, latency, cost estimates, and failure states.

README Run Contract

1. Clone
2. Install
3. Copy .env.example
4. Seed demo data
5. Run app
6. Run tests
7. Run eval smoke test
8. Inspect traces/report

Minimum CI Gate

  • Typecheck.
  • Unit tests.
  • Schema validation tests.
  • Tool policy tests.
  • Small eval smoke test.
  • Secret scanning or documented equivalent.

Interactive Drill: Good Portfolio Ops?

The repo has seed data and a saved trace, so the reviewer can inspect behavior without private docs.

The demo requires a personal API key, three manual DB steps, and undocumented fake users.

CI blocks a change that breaks structured output validation.

Build Assignment

Add a deployment/reproducibility pass to the portfolio.

  • Create .env.example with safe placeholder values.
  • Add seed data and a demo mode.
  • Add a documented run path using local commands or Docker.
  • Add a health endpoint or debug panel.
  • Add CI for tests and a small eval smoke test.
  • Write docs/deployment-notes.md with tradeoffs and known limits.

Ask the agent to try the README as a skeptical reviewer and list every unclear setup step.
