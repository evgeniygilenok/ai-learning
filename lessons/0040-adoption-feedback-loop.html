Lesson 0040 - 35 minutes

Use feedback to prove product judgment.

A portfolio gets stronger when real people try it. The skill is turning feedback into scoped fixes, eval cases, and better interview stories.

The One Idea

External feedback is not a popularity test. It is a way to discover where the AI product contract is unclear, brittle, slow, unsafe, or unconvincing.

Recruit

Ask 3-5 engineers, product people, or domain-adjacent users to try one specific workflow.

Observe

Watch setup friction, confusing states, trust breakdowns, wrong answers, and missing explanation.

Convert

Turn feedback into issues, eval cases, docs, or product changes.

Close

Fix the top three issues and record what changed in the demo, README, or quality report.

Feedback Prompt

Please try the knowledge assistant flow:
1. Ask a policy question.
2. Inspect the citations.
3. Try a question it should refuse.
4. Tell me where trust broke.
5. Tell me what would make this easier to review.

Feedback Record

source: senior backend engineer
workflow: RAG answer with citations
issue: citations were present but not easy to inspect
contract affected: quality/safety
fix: add source preview and trace link
new eval: citation must reference expected section

Interactive Drill: What Should This Become?

"It answered confidently when the docs did not contain the policy."

"I could not tell how to run the demo without your help."

"The approval gate made me trust the agent more, even though it slowed the flow."

Build Assignment

Create docs/adoption-feedback.md.

  • Ask 3-5 people to try one workflow.
  • Record what they tried, where they got stuck, and where trust broke.
  • Classify each note as UX, quality, security, setup, performance, or explanation.
  • Fix the top 3 issues.
  • Add at least one new eval case from feedback.

Ask the agent to turn raw feedback into a prioritized issue list.
