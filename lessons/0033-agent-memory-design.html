Lesson 0033 - 35 minutes

Memory is a product policy, not a bigger prompt.

Memory design decides what should survive beyond a run, why it helps, how it is retrieved, and when it must be hidden or deleted.

The One Idea

Agent memory is useful only when it has a job. It can improve personalization, project continuity, and workflow recovery, but it also expands privacy, security, and correctness risk.

The design move is to separate memory types, then write save and retrieve rules for each type. Do not let every transcript fragment become future context.

Working Memory

Short-lived run context that helps the current task. It belongs in state and expires quickly.

Preference Memory

Explicit user choices that should shape future outputs, such as format, detail level, or review style.

Project Memory

Stable team or repository facts: owners, conventions, architecture decisions, and trusted docs.

Episodic Memory

Past interaction summaries that help continuity, but should be compact, sourced, and easy to inspect.

Audit History

Durable action records for accountability. These may inform debugging, but should not be blindly injected into prompts.

Forbidden Memory

Secrets, transient credentials, sensitive inferred traits, stale guesses, and unverified facts.

Memory Record Shape

type AgentMemory = {
  id: string;
  scope: "user" | "project" | "tenant" | "audit";
  kind: "preference" | "fact" | "episode" | "action_log";
  content: string;
  source: "explicit_user" | "approved_summary" | "tool_result" | "system_event";
  visibility: "private" | "team" | "admin";
  confidence: "verified" | "inferred";
  expiresAt?: string;
};

Memory Gates

  • Save only with a named future use.
  • Prefer explicit user statements over inference.
  • Attach source, visibility, and expiration.
  • Retrieve only memories relevant to the current task and actor.
  • Give users a way to inspect and correct important memories.

Interactive Drill: Save, Summarize, Or Reject?

The user says, "For this repo, use Vitest examples unless I ask otherwise."

A 40-message debugging session ends with three stable repo conventions and many dead ends.

A stack trace includes a temporary access token from a failed tool call.

Privacy Habit

Memory needs deletion, correction, and visibility rules from the beginning. Otherwise a helpful feature becomes a quiet data retention system.

Build Assignment

Create a memory policy matrix for the coding workflow agent:

  • Five allowed memories with scope, source, visibility, and expiration.
  • Five forbidden memories with the reason each is blocked.
  • One retrieval rule for each workflow state.
  • One user-facing control for correcting or deleting memory.

Ask the agent to review the matrix for over-retention and prompt-injection risk.
