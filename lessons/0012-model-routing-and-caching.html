Lesson 0012 - 30 minutes

Route expensive intelligence only where it earns its keep.

Model routing and caching turn latency and cost from vague complaints into explicit product controls.

The One Idea

Not every request needs the same model, context, or generation path. A production AI app can classify work, choose the cheapest sufficient path, and escalate only when quality or risk requires it.

Caching is the partner move: reuse stable work such as retrieval results, policy summaries, deterministic tool responses, and repeated prompt context when it is safe to do so.

Fast Path

Use a smaller model or deterministic rule for low-risk classification, routing, and simple extraction.

Standard Path

Use the default model and retrieval setup for normal product triage or knowledge answers.

Escalation Path

Use a stronger model, stricter validation, or human review for high-risk or ambiguous work.

Fallback Path

Use cached results, degraded answers, or background jobs when providers are slow or unavailable.

Routing Policy Shape

type RouteDecision = {
  taskType: "classify" | "triage" | "rag_answer" | "workflow_plan";
  risk: "low" | "medium" | "high";
  latencyBudgetMs: number;
  costBudget: "low" | "standard" | "premium";
  model: "small" | "default" | "strong";
  cacheMode: "read" | "write" | "bypass";
  requiresReview: boolean;
};

What Can Be Cached?

  • Embedding results for unchanged chunks.
  • Read-only tool responses with a clear freshness window.
  • Retrieval result IDs for repeated queries.
  • Stable system prompt or policy context where supported.
  • Never cache sensitive output without a retention policy.
